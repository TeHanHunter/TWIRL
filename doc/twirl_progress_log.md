- `2026-04-14`: Diagnosed and fixed a severe thread-oversubscription performance bug in detrend. Root cause: each multiprocessing worker silently spawned BLAS/OpenMP threads equal to core count, so `-n 8` on `pdogpu1` (128 cores) became `~1000` threads (load avg `303`, ~4 lc/min effective). Fix: set `OMP_NUM_THREADS=OPENBLAS_NUM_THREADS=MKL_NUM_THREADS=VECLIB_MAXIMUM_THREADS=NUMEXPR_NUM_THREADS=BLIS_NUM_THREADS=1` at module load in [detrend_wrapper.py](../scripts/detrend_wrapper.py) (before numpy import) and bump to `-n 32`. Result: `~200×` speedup (`16 h` → `5 min` for a full orbit). Thread caps are now also baked into [hlsp_wrapper.py](../scripts/hlsp_wrapper.py). CLAUDE.md PDO-Specific Notes now documents this as a standing rule for any PDO multiprocessing pool.
- `2026-04-14`: Tightened fast-skip in [detrend_wrapper.py](../scripts/detrend_wrapper.py) to make re-runs correct and O(1). Initial version only checked `bestdmagkey` attr existence, which wrongly skipped LCs from a crashed prior run that had set the attr without writing the corresponding magnitude dataset. Updated check verifies `PrimaryAperture[bestdmagkey]` dataset is actually present; otherwise returns `False` and re-detrends. This check is necessary because `qlp.lctools.bin.detrend.detrend_qsp_h5` does not short-circuit on already-detrended inputs.
- `2026-04-14`: Identified cam3/orbit-120 failure mode as a real TGLC/QLP data-alignment bug, not a code issue. Caught by the safe wrapper and logged as `[detrend skip]`. These targets make up the entire `22%` HLSP shortfall. The bug is deterministic per-CCD (all of cam3 fails, cam1/2/4 all pass), implying an off-by-one in how the cam3 quaternion file's cadence range aligns with the TGLC lightcurve BJD grid.

**Next:** Reconcile the cam3/orbit-120 `6137` vs `6136` cadence mismatch between TGLC BJD and QLP quaternion features (check the cam3 quat file cadence range against orbit-120 TICA FFI cadences; the off-by-one is likely at the orbit boundary). After fix: re-run detrend + HLSP for cam3/orbit-120 only, then proceed to WD 1856 benchmark QA (Stage 1.6–1.7).
