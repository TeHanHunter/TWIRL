# Reports

`reports/` contains compact QA, benchmark, analysis, and handoff evidence
generated by the pipeline. It is organized by the Stage 1–5 structure, with
`reports/exploratory/` reserved for work that is not yet an accepted pipeline
result.

## Authority And Freshness

A report records what was true for its named inputs, code state, and capture
date. It is **not** a live project plan. In particular, a `plan.md`,
`status.md`, `summary.md`, or JSON status file inside a report directory may be
superseded even when the artifact remains useful evidence.

Use the documentation hierarchy in this order:

1. [twirl_plan.md](../doc/twirl_plan.md) for current decisions, gates, and
   forward priorities;
2. [twirl_progress_log.md](../doc/twirl_progress_log.md) for dated execution
   history and benchmark results;
3. the relevant operational or methods document under [doc](../doc/);
4. report-local prose and machine-readable summaries for the captured run.

The presence of a report directory does not by itself mean a branch, sector,
model, or candidate has been accepted. Check the live plan and the report's
validation fields before reusing it.

## Layout

- `stage1_lightcurves/`: catalog, extraction, A2v1, photometric, and production
  QA
- `stage2_search/`: transparent-search benchmarks and detection diagnostics
- `stage3_injections/`: injection construction and recovery evidence
- `stage5_validation/`: vetting, labeling, model, candidate, and validation
  artifacts; this currently also contains transitional S56 search/training
  experiments
- `exploratory/`: talks, one-off investigations, and non-authoritative plots

`stage4_search/` should be added when frozen full-survey inference begins. Do
not relabel an active-learning pilot as a Stage 4 result merely because it
scores many candidates.

## Report Expectations

Reusable reports should record, when applicable:

- capture time and sector/product tag;
- input paths or stable product identifiers;
- configuration and aperture/flux-column contract;
- code revision or reproducible script entrypoint;
- sample size, exclusions, failures, and acceptance-gate result;
- links to the corresponding progress-log entry.

Keep large light curves, raw catalogs, model checkpoints, full vetting-sheet
collections, and machine-local staging outside git. Commit compact summaries,
figures, manifests, or representative samples only when they are reusable.
Generated report artifacts do not replace provenance in the underlying FITS,
HDF5, JSON, or catalog metadata.

Use "candidate retention" for partial chains such as BLS-to-teacher audits.
Reserve "end-to-end completeness" for a frozen measurement that includes
extraction calibration, search, any production ranker, automated vetting, and
candidate merging over the declared parent sample.
